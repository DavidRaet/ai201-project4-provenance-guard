## Project Overview

**Provenance Guard** is a backend classification system that analyzes submitted creative text and determines whether it is likely AI-generated or human-authored. It returns a structured attribution result, confidence score, and a plain-language transparency label. Creators can appeal classifications, and all decisions are recorded in a structured audit log.

**Tech Stack:**
- **Framework:** Flask
- **LLM Signal:** Groq (llama-3.3-70b-versatile)
- **Stylometric Heuristics:** Pure Python (type-token ratio, sentence length, punctuation, function words)
- **Rate Limiting:** Flask-Limiter (per-IP address)
- **Dependencies:** See `requirements.txt`

## Architecture

### Three-Tier Detection System

**Signal 1: LLM-Based Semantic Analysis (Groq)**
- Function: `llm_signal(text: str) -> float`
- Analyzes structural and semantic patterns typical of AI text
- Output: probability score in `[0.0, 1.0]` (1.0 = high confidence AI)
- Weight: 0.65 in the final confidence score

**Signal 2: Stylometric Heuristics (Pure Python)**
- Function: `stylometric_signal(text: str) -> float`
- Features: type-token ratio, avg sentence length, punctuation density, function word frequency, filler phrase detection
- Output: normalized score in `[0.0, 1.0]`
- Weight: 0.35 in the final confidence score

**Signal Combination:**
- Function: `combine_signals(s1: float, s2: float) -> float`
- Formula: `confidence_score = (0.65 * signal_1) + (0.35 * signal_2)`
- Output: final score in `[0.0, 1.0]`

### Confidence Thresholds & Labels

Confidence scores are bucketed into three ranges, each with a plain-language transparency label:

| Score Range | Classification | Label Key | Label Text |
|---|---|---|---|
| 0.80–1.00 | High-confidence AI | `AI_HIGH` | "This content was likely generated by an AI tool." |
| 0.40–0.79 | Uncertain | `UNCERTAIN` | "We're not sure who created this." |
| 0.00–0.39 | High-confidence Human | `HUMAN_HIGH` | "This content appears to be human-written." |

### Audit Log

All submissions and appeals are recorded in a structured audit log as JSON entries. Each entry includes:
- `content_id`: unique identifier for the submission
- `timestamp`: when the submission was classified
- `classification` or `appeal`: the type of entry
- For classifications: `score`, `confidence_score`, `label_key`
- For appeals: `status` update (usually "under_review"), `reasoning` from the creator
- All previous entries remain immutable

## API Endpoints

### POST /submit
- **Rate limit:** 10 requests per minute (per IP)
- **Request body:** `{"text": "..."}` 
- **Response:** JSON with `content_id`, `confidence_score`, `label_key`, `label_text`, `label_detail`
- **Validation:** Text must be non-empty
- **Flow:** Text → Groq signal + Stylometric signal → combine → threshold bucket → JSON response + log

### POST /appeal
- **Rate limit:** 5 requests per minute (per IP)
- **Request body:** `{"content_id": "...", "reasoning": "..."}`
- **Response:** JSON confirmation
- **Validation:** 
  - `reasoning` must be present and non-empty (400 Bad Request if missing)
  - `content_id` must exist in log (404 Not Found if not)
  - Cannot appeal same `content_id` twice while status is "under_review" (409 Conflict)
- **Flow:** Validate inputs → update content status to "under_review" → append appeal to log

### GET /log
- **Rate limit:** 30 requests per minute (per IP)
- **Response:** JSON array of all audit log entries (chronological order)
- **No parameters**
- **Flow:** Return structured log entries, no processing

## Implementation Phases

The implementation is split into three milestones, each adding a discrete piece:

### Milestone 3: Flask App + LLM Signal
- Skeleton Flask app with `/submit` endpoint
- `llm_signal()` function with Groq API call
- `combine_signals()` placeholder (s2 = 0.0 until M4)
- **Validation:** Run `llm_signal()` on three test cases and verify scores land in expected zones

### Milestone 4: Stylometric Signal + Full Scoring
- `stylometric_signal()` with all heuristics
- Updated `combine_signals()` using both signals at weights 0.65/0.35
- `label_selector()` that maps score to label key
- **Validation:** Full pipeline on same three test cases; verify distinct labels for clearly different inputs

### Milestone 5: Production Layer (Labels, Appeals, Audit Log)
- `format_label()` returning plain-language text per spec
- `POST /appeal` endpoint with all validations
- `GET /log` endpoint
- **Validation:** Hit all three endpoints; confirm appeals show status and reasoning; verify duplicate-appeal guard

## Key Design Decisions

1. **Rate limiting by IP:** No authentication layer in v1, so IP is the practical choice. Trade-off: users behind shared NATs may be unfairly grouped.

2. **Fixed-window rate limiting:** Flask-Limiter's default algorithm. Simpler than token-bucket or sliding-window.

3. **Audit log as source of truth:** Both submissions and appeals are appended; content status is updated in place. All history is preserved.

4. **Weighted average for signal combination:** LLM signal (0.65) weighted higher because it captures deeper semantic structure; stylometrics (0.35) serve as a fast, explainable cross-check. Weights may be tuned after validation testing.

5. **Three-label bucketing (not continuous scores):** Thresholds at 0.40 and 0.80 provide clear high-confidence regions and an explicit "uncertain" zone. Avoids false precision of raw scores.

6. **Appeals do NOT trigger re-classification:** Appeals validate the target, update status, and log the reasoning. They are a human-review signal, not a re-run of the detection pipeline.

## Testing & Validation

### Pre-Deployment Testing Approach

Before each milestone is considered complete:

1. **Functional tests:** Hand-test each endpoint with curl or a script; verify response structure and status codes.

2. **Signal validation tests:** For M3 and M4, test `llm_signal()` and `stylometric_signal()` directly on:
   - A clearly AI-generated passage (expected score > 0.80)
   - A personal essay excerpt (expected score < 0.35)
   - Ambiguous mixed-voice text (expected 0.40–0.70)

3. **Label consistency:** Confirm that clearly different inputs produce different labels (not all "uncertain").

4. **Rate-limit verification:** After M5, confirm that rate limits return 429 with `Retry-After` header.

5. **Appeal edge cases:** Confirm missing `reasoning` → 400, missing `content_id` → 404, duplicate appeal → 409.

### Caveat
The system relies on the Groq LLM's quality. If the prompt or model behavior changes, signal scores may drift. Re-validate thresholds after any prompt tweaks.

## Common Pitfalls & Gotchas

1. **Groq API rate limits:** Groq's free tier has its own rate limits. 10 submissions/minute times (N users) can hit Groq limits. Monitor and adjust app limits if needed.

2. **GROQ_API_KEY missing:** The app will fail to initialize if the key is not set. Always include `.env` setup in new-environment instructions.

3. **Floating-point precision:** Stylometric scores are sums of normalized features. Ensure the final score stays in [0.0, 1.0] (clamp if needed).

4. **Appeal-then-resubmit:** If a creator appeals and later resubmits the same or similar text, it will get a new `content_id` and be re-classified. This is by design (each submission is independent), but creators may be confused.

5. **Audit log size:** As submissions and appeals accumulate, the log grows unbounded. For a production system, consider pagination or archival.
